## Plan: refactor `ApprovalService.process()` into smaller methods

**Target shape.** `process()` stays the only public `@Transactional` entry point (~12 lines) and reads as the pipeline narration; all extracted methods are **private on the same bean** (moving them to another bean or splitting the transaction boundary would change semantics via Spring self-invocation).

**Extraction steps** (one at a time, `./mvnw test` after each — all are pure "Extract Method" moves, no logic edits):

1. **`validateProcessable(request)`** — owns the guards (lines 59–65): null → `IllegalArgumentException`, non-`SUBMITTED` → `IllegalStateException`.
2. **`denyForEligibility(request, eligibility)`** — owns the Stage-1 denial short-circuit (lines 72–82): set `DENIED`, stamp `decidedAt`, save request, create the `Denial` from the *saved* request (preserve that ordering), notify, log, return.
3. **`logHeavyDenialHistory(patientRef)`** — owns Stage 2 (lines 88–95): fetch history, count `"DENIED"`, log when > 3. Void return makes its advisory, non-gating nature explicit.
4. **`applyThresholdDecision(request)`** — owns Stage 3 (lines 98–107): `AUTO_APPROVED` vs `MANUAL_REVIEW` plus logging. Copy the comparison **verbatim** (see flag #2). The shared `decidedAt` stamp + final save stay in `process()`.
5. Optional: a tiny shared `stampDecided(request)` reusable from `resolveManualReview()`. Skip a shared "deny + notify" helper — the two denial paths save the request and the `Denial` in opposite orders, so unifying them isn't a pure extraction.

**Test impact.** Zero required changes: all 10 `ApprovalServiceTest` tests drive the public API against mocks, and `PriorAuthControllerTest` mocks `ApprovalService` entirely — they're the safety net, not casualties. Don't widen helper visibility to test private methods. One gap worth closing *before* refactoring: no test pins `score == 0.85` exactly; add a characterization test asserting today's behavior (`0.85` → `MANUAL_REVIEW`) so an accidental boundary change fails loudly.

**Flags against the refactor:**

1. **The code explicitly says don't.** The Javadoc on `process()` reads: *"this method is intentionally long-form… Refactoring it is a course lab — resist tidying it."* The README confirms this is a teaching repo where labs branch from `lab/start` — the current branch. If this is the intended lab, do it on a fresh branch off `lab/start` (other labs branch from it); if it's not the lab, that comment is a hard no.
2. **Latent threshold bug the refactor could silently change.** The class doc and the approval log line both say the threshold is inclusive (`>= 0.85` approves), but the code uses strict `score > AUTO_APPROVE_THRESHOLD` — so `0.85` currently routes to manual review. No test covers the boundary. The refactor must preserve `>` bit-for-bit; fixing the discrepancy is a separate decision/commit.
3. **Modest payoff.** ~50 lines, one production caller (`PriorAuthController.java:54`), real logic already delegated to `EligibilityService`/`ClaimHistoryService`, and clear stage-banner comments — extraction mostly turns those comments into method names.
4. **Transaction constraints** rule out fancier shapes (stage beans, public step methods).

Verification: boundary test green pre-refactor, full suite green after each step with no test modified, and a final `git diff` showing only `ApprovalService.java` plus the one new test.
