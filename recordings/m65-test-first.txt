THE MECHANISM (verified in the test source, not assumed):
  The TDD tests express the spec in LITERAL DOMAIN TERMS —
    queued(Duration.ofDays(days)) · Duration.ofDays(2).plusHours(23) · ofDays(5)
  and NEVER reference AGING_AFTER_DAYS / BREACHED_AFTER_DAYS.
  So when a constant moves, the test still demands the SPECIFIED behavior and fails.
  Contrast M-5.2's retrofitted boundary tests, which referenced
  ApprovalService.AUTO_APPROVE_THRESHOLD and therefore FOLLOWED the constant —
  which is exactly why the 0.85->0.84 mutation survived there.
  Test written FIRST from the spec  -> asserts in spec terms -> pins operator AND value.
  Test written AFTER from the code  -> reaches for the code's constants -> pins the
                                       operator at best, never the value.

HONEST SCOPE: this is one recorded pair (n=1), not proof that TDD always scores
higher. What IS demonstrated is the MECHANISM above, and it is reproducible:
write the assertion in the language of the requirement, not of the implementation.
